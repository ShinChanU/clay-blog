## Value Proposition (6-Part JTBD Template)

### 1. WHO — 누구를 위한 사이트인가

프론트엔드 개발자를 채용하려는 **스타트업/중견기업의 채용담당자와 기술 면접관**.
특히 "이력서만으로는 지원자의 실력을 판단하기 어렵다"고 느끼는 사람.

### 2. WHY — 왜 이 사이트가 필요한가

**Before State (현재 문제):**
- 이력서에 기술 나열만 있고, "왜 그 기술을 선택했는지" 판단할 수 없음
- 비유명 회사 + 3~4년차 = 서류에서 걸러지기 쉬운 구간
- AI가 쓴 것 같은 완벽한 이력서 → 진짜 실력인지 의심

**Burning Need:**
> "이 지원자가 실제로 어떤 수준의 문제를 풀어봤는지, 이력서 너머의 증거가 필요하다"

### 3. WHAT BEFORE — 현재 대안은 무엇인가

| 대안 | 한계 |
|------|------|
| 이력서만 보기 | 깊이를 알 수 없음. 비유명 회사 출신은 기대치 설정 불가 |
| GitHub 확인 | 코드는 있지만 "왜"가 없음. 맥락 없는 코드는 판단이 어려움 |
| 일반 기술 블로그 | 튜토리얼/학습 정리 수준이면 실무 능력 판단 불가 |
| 전통적 포트폴리오 | 스킬 나열 + 프로젝트 카드 = 모든 개발자가 비슷해 보임 |

### 4. HOW — 이 사이트는 어떻게 다른가

**핵심 차별화: "판단과 맥락이 드러나는 포트폴리오"**

| 일반 개발자 포트폴리오 | claychanwoo.com |
|----------------------|----------------|
| "React, TypeScript, Next.js 사용" | "Webpack→Vite를 점진적으로 전환한 이유와 결과" |
| "대시보드 개발" | "50개 카드 타입 아키텍처를 어떤 트레이드오프로 설계했는지" |
| "디자인 시스템 구축" | "디자이너가 실제로 쓰는 시스템을 만들기 위해 Figma API를 자동화한 과정" |
| "성능 최적화" | "800초→33초, 번들 80%↓, 로딩 1/4 — 구체적 수치와 방법론" |

### 5. WHAT AFTER — 사이트를 본 후 어떤 상태가 되는가

**After State:**
- 채용담당자: "이 사람은 면접에서 더 얘기해볼 가치가 있다" → 면접 제안
- 테크리드: "기술적 사고가 논리적이고, 면접에서 이 부분을 파보면 좋겠다" → 질문 준비 완료
- 동료 개발자: "실무 경험이 잘 정리된 블로그. 구독해야겠다" → RSS/북마크

### 6. ALTERNATIVES — 경쟁 포트폴리오 대비 왜 이 사이트인가

**Unfair Advantage (대체 불가능한 강점):**
1. **사이드 프로젝트 실제 운영 지표** — MAU 156% 증가, 검색 1위, 누적 1,100명 가입. 대부분의 개발자 포트폴리오에는 "만들었다"만 있지 운영 지표가 없음.
2. **"사이드→본업" 학습 루프의 증거** — FSD를 사이드에서 실험 후 본업에 적용한 궤적. 단순 사이드가 아닌 "체계적 학습자"임을 증명.
3. **에이전트 코딩 전문성** — Cursor Skills 30개 설계, Spec 주도 개발. AI 시대의 개발 생산성을 실제로 시스템화한 경험.

## Positioning — 차별화 포인트

### 현재 경쟁 환경 분석

한국 프론트엔드 개발자 포트폴리오의 전형적 패턴:
```
[스킬 뱃지 나열] + [프로젝트 카드 그리드] + [경력 타임라인] + [연락처]
```

**문제**: 모든 포트폴리오가 이 구조 → 차별화 안 됨 → 결국 회사 이름과 학력으로 판단됨

### 포지셔닝 전략

**포지셔닝 문장:**
> "기술을 나열하지 않고, **판단의 과정**을 보여주는 개발자 포트폴리오"

**차별화 축:**

| 축 | 대부분의 포트폴리오 | claychanwoo.com |
|----|-------------------|----------------|
| **콘텐츠 깊이** | "했다" 중심 | "왜 그렇게 했고, 어떤 대안이 있었는지" 중심 |
| **증거 유형** | 기술 스택 태그 | 정량 수치 (800초→33초, MAU 156%↑) |
| **톤** | 객관적/형식적 | 실제 경험에서 우러나온 솔직한 회고 |
| **구조** | 스킬→프로젝트→경력 순서 | 페르소나별 경로 분리 (Projects / Blog / About) |

### 태그라인 후보

1. **"비효율을 시스템으로 바꾸는 프론트엔드 개발자"** ← 커리어 전략의 핵심 강점 직결
2. "코드로 문제를 구조화하는 프론트엔드 개발자"
3. "만들고, 측정하고, 개선하는 프론트엔드 개발자"

**추천: 1번** — 가장 구체적이고, 사이트 전체 콘텐츠(자동화, 시스템화, 표준화 사례)와 일관됨

## 사이트 메시지 아키텍처

Discovery 결과를 사이트 각 영역의 메시지로 변환:

### 랜딩 Hero
- **메인 카피**: "비효율을 시스템으로 바꾸는 프론트엔드 개발자"
- **서브 카피**: "자동화, 아키텍처 설계, 그리고 실제 유저가 쓰는 서비스를 만듭니다"
- **CTA**: "프로젝트 보기" / "기술 블로그"

### Featured Projects (랜딩 섹션)
- 각 카드에 **수치 한 줄** 필수:
  - "디자인 토큰 자동화 96% 단축"
  - "MVP 5개월 출시, 누적 1,100명"
  - "번들 80%↓, 로딩 1/4"

### 블로그
- **관점 선언** (kilee 패턴): "기술 선택의 '왜'를 기록합니다"
- 글 제목에 의사결정 키워드 포함: "~를 선택한 이유", "~에서 배운 것", "~마이그레이션기"

### About
- **서사 구조**: 비유명 회사 + 비서울권 대학의 핸디캡을 인정하되, "그래서 사이드에서 검증하고 본업에 가져오는 루프를 만들었다"는 스토리
- **수치 요약**: 핵심 지표 5~7개를 시각적으로 표현

## 메시지 일관성 체크리스트

사이트 전체에서 아래 4가지 메시지가 반복 강화되어야 함:

- [ ] **"반복→시스템"**: 자동화/표준화 사례가 프로젝트와 블로그에 일관되게 등장
- [ ] **"0→1 실행력"**: MVP 출시, 사이드 프로젝트 운영 지표가 Featured에 노출
- [ ] **"사이드→본업 루프"**: About과 블로그에서 "실험 후 본업 적용" 패턴이 보임
- [ ] **"왜(Why) 중심"**: 케이스 스터디와 블로그 글 모두 의사결정 과정 중심 서술
