# Ideal Customer Profile (ICP)

> 이 사이트의 "고객"은 누구인가?
> claychanwoo.com은 제품이 아닌 **개인 브랜딩 사이트**이므로, 고객 = "이 사이트를 보고 판단을 내리는 사람"

---

## Primary ICP: 스타트업/중견기업 채용담당자

### Demographics
- **회사 규모**: 50~300명 스타트업 또는 중견 테크기업
- **산업**: SaaS, 핀테크, 커머스, AI/ML 스타트업
- **직함**: HR 매니저, 채용 담당자, 피플팀 리드
- **경험**: 기술 채용 1~5년차
- **특성**: 하루에 이력서 20~50개를 검토하며, 한 이력서에 평균 30초~2분 투자

### Behaviors
- 이력서의 "링크" (블로그, GitHub, 포트폴리오)를 클릭해서 30초 안에 Pass/Fail 판단
- 기술 깊이보다는 **"이 사람이 우리 팀에서 잘 일할 수 있을까?"**를 직감적으로 판단
- 회사 이름, 학력을 먼저 확인 → 모르는 회사면 포트폴리오/블로그로 보완 검증
- 시각적 첫인상에 크게 좌우됨 (잘 만든 사이트 = 능력 있는 개발자라는 무의식)

### JTBD (Jobs to Be Done)
- **Primary Job**: "이 지원자가 팀에 기여할 수 있는 사람인지 30초 안에 판단하고 싶다"
- **Emotional Job**: "뽑았는데 안 맞으면 내 평가에 영향 → 안전한 선택을 하고 싶다"
- **Social Job**: "좋은 인재를 빠르게 찾아서 팀/경영진에게 인정받고 싶다"

### Pain Points
1. 이력서만으로는 "이 사람이 실제로 이걸 했는지" 확인이 안 됨
2. 비유명 회사 + 비서울권 대학 → 기대치 설정이 어려움
3. 3~4년차의 애매한 구간 → 주니어? 시니어? 판단이 안 서는 불편함
4. 기술 나열형 이력서 → AI가 쓴 것 같은 느낌 → 신뢰도 하락

### 이 ICP가 사이트에서 원하는 것
- **3초**: "이 사람이 프론트엔드 개발자구나" + "뭔가 잘 만들었네" (시각적 신뢰)
- **30초**: "사이드 프로젝트 실제 지표가 있네" + "기술 글을 쓰는 사람이네" (깊이 증거)
- **2분**: 프로젝트 케이스 스터디 1개 확인 → "면접에서 이 얘기를 물어봐야지"

## Secondary ICP: 기술 면접관 / 테크리드

### Demographics
- **직함**: 시니어 개발자, 테크리드, 프론트엔드 챕터 리드, CTO
- **경험**: 개발 5~15년차
- **특성**: 기술적 깊이를 직접 평가할 수 있는 사람. 이력서보다 코드와 글을 본다.

### Behaviors
- 이력서 링크의 **블로그 글**을 실제로 읽어봄 (제목만 보고 깊이 판단)
- GitHub 잔디보다 **PR 내역, 커밋 메시지 품질, 아키텍처 결정**을 봄
- "왜 이 기술을 썼는지" 설명할 수 있는지가 가장 중요한 판단 기준
- 면접 질문 소재를 사이트에서 뽑아감

### JTBD
- **Primary Job**: "이 사람의 기술적 사고 수준을 빠르게 파악해서 면접 전에 기대치를 설정하고 싶다"
- **Emotional Job**: "면접 시간을 낭비하고 싶지 않다"
- **Social Job**: "내가 추천한 후보가 합격하면 팀에서의 신뢰가 올라간다"

### Pain Points
1. 기술 나열만 있고 "왜"가 없으면 실력 판단이 안 됨
2. 블로그에 튜토리얼만 있으면 실무 수준 판단 불가
3. GitHub에 사이드 프로젝트만 있고 실무 코드가 없으면 아쉬움

### 이 ICP가 사이트에서 원하는 것
- **블로그**: "Webpack→Vite 마이그레이션 결정 과정" 같은 의사결정 글
- **프로젝트**: 아키텍처 다이어그램 + 트레이드오프 설명
- **수치**: "800초→33초" 같은 정량 결과

## Tertiary ICP: 동료 개발자 / 네트워크

### Demographics
- **직함**: 주니어~시니어 프론트엔드 개발자
- **특성**: 기술 블로그를 탐색하다가 유입, 또는 SNS/커뮤니티 공유로 유입

### Behaviors
- 검색(구글/네이버)으로 기술 글에 직접 랜딩
- 좋은 글이면 저자 프로필 확인 → 다른 글도 탐색
- RSS/Twitter 팔로우 → 장기적 네트워크

### JTBD
- **Primary Job**: "내가 겪고 있는 기술 문제의 해답을 찾고 싶다"
- **Secondary Job**: "괜찮은 개발자를 발견하면 북마크해두고 싶다"

### 이 ICP가 사이트에서 원하는 것
- **SEO가 잘 된 기술 글** (검색으로 바로 유입)
- **실무 기반 깊이 있는 콘텐츠** (튜토리얼이 아닌 경험담)
- **RSS 피드** (구독 경로)

## ICP 우선순위 매트릭스

| ICP | 사이트 체류시간 | 핵심 판단 기준 | 전환 목표 | 우선순위 |
|-----|----------------|---------------|----------|---------|
| 채용담당자 | 30초~2분 | 시각적 신뢰 + 수치 증거 | 면접 제안 | **1순위** |
| 테크리드 | 5~15분 | 기술적 사고 깊이 | 면접 합격 추천 | **2순위** |
| 동료 개발자 | 3~10분 | 콘텐츠 품질 | 구독/공유 → 간접 추천 | 3순위 |

---

## Disqualification (이 사이트의 고객이 아닌 사람)

- 대기업 HR의 자동 ATS 스크리닝 (사이트를 보지 않음)
- 순수 디자이너/PM (기술 콘텐츠에 관심 없음)
- 코딩 테스트만으로 평가하는 회사 (포트폴리오 불필요)

---

## 핵심 시사점

1. **첫인상 3초가 전부다** — 채용담당자가 1순위이므로, 랜딩 페이지의 시각적 품질이 생사를 가름
2. **깊이는 서브페이지에서** — 테크리드가 2순위이므로, 블로그와 케이스 스터디의 깊이가 면접 확률을 결정
3. **수치가 신뢰를 만든다** — "했다"가 아닌 "800초→33초", "MAU 156%↑" 같은 정량 데이터가 세 ICP 모두에게 통함
4. **"왜"가 차별화다** — 기술 나열이 아닌 의사결정 과정이 테크리드를 설득하는 핵심
