# Customer Journey Map — 채용 동선 설계

> 이력서 링크 클릭부터 면접 결정까지, 각 터치포인트에서의 경험 설계
> Primary Persona (채용담당자 김서연) 중심, Secondary (테크리드 박준혁) 병행

---

## Journey Overview

```
[이력서 확인] → [사이트 첫 방문] → [탐색] → [깊이 확인] → [판단/결정]
   0초            0~3초           3~30초      30초~5분        5분+
```

---

## Stage 1: 이력서 확인 (사이트 진입 전)

### 터치포인트
- 원티드/사람인/잡코리아에서 이력서 열람
- 이력서 하단의 "블로그: claychanwoo.com" 링크 발견

### 감정/생각
- 김서연: "다키클라우드? 에너지엑스? 잘 모르는 회사네... 일단 블로그라도 봐볼까"
- 박준혁: "사이드 프로젝트가 많네. 블로그 글 수준을 봐야겠다"

### 이탈 위험
- 이력서 자체가 매력 없으면 링크를 클릭하지 않음
- **대응**: 이력서에 "claychanwoo.com — 기술 블로그 & 프로젝트 포트폴리오" 명시

### 핵심 목표
- 이력서에서 사이트 링크를 **반드시 클릭하게** 만들기

---

## Stage 2: 사이트 첫 방문 — First Impression (0~3초)

### 터치포인트
- claychanwoo.com 랜딩 페이지 (/)

### 감정/생각
- 김서연: "오, 깔끔하네" 또는 "음... 좀 별로인데?" (3초 안에 결정)
- 박준혁: "개발자가 직접 만든 사이트네. 기술력이 느껴지나?"

### 이 순간에 전달해야 할 것 (3초 내)
1. **누구인가**: 이름 + "프론트엔드 개발자" (명확한 포지셔닝)
2. **뭘 하는 사람인가**: 한 줄 태그라인 (예: "비효율을 시스템으로 바꾸는 프론트엔드 개발자")
3. **시각적 신뢰**: 잘 만들어진 사이트 = 실력 있는 개발자

### 이탈 위험 (높음 — 가장 위험한 구간)
- 로딩이 2초 이상 걸리면 → 즉시 이탈
- 모바일에서 깨지면 → "프론트엔드 개발자인데 반응형도 안 돼?" → 즉시 이탈
- 디자인이 촌스러우면 → 무의식적 실력 의심 → 이탈
- 콘텐츠가 텅 비어 있으면 → "블로그에 글이 없네" → 이탈

### 핵심 목표
- **스크롤을 유도**하거나 **다음 페이지 클릭**을 유도

---

## Stage 3: 탐색 — Scanning (3초~30초)

### 터치포인트
- 랜딩 페이지 스크롤 (Featured Projects, Recent Posts 영역)
- Navbar 메뉴 확인

### 감정/생각
- 김서연: "프로젝트가 있네... MAU 156% 증가? 괜찮은데. 하나 클릭해볼까"
- 박준혁: "블로그 글 제목이... Webpack→Vite? 어댑터 패턴? 주제가 괜찮네"

### 이 순간에 전달해야 할 것
1. **수치 기반 프로젝트 카드**: 썸네일 + 제목 + 핵심 성과 수치 1줄
2. **블로그 글 미리보기**: 제목만으로 기술 깊이가 느껴져야 함
3. **명확한 네비게이션**: Projects / Blog / About — 다음 행동 경로가 분명

### 이탈 위험
- Featured Projects 카드가 "기술 스택만 나열"되어 있으면 → 차별화 안 됨 → 이탈
- 블로그 글이 2~3개뿐이면 → "콘텐츠가 별로 없네" → 이탈

### 핵심 목표
- 김서연 → /projects/[slug] 클릭
- 박준혁 → /blog 또는 /blog/[slug] 클릭

---

## Stage 4: 깊이 확인 — Deep Dive (30초~5분)

### 터치포인트 A: 프로젝트 케이스 스터디 (/projects/[slug])
- **김서연 경로**
- 프로젝트 개요 → 내 역할 → 문제-해결-결과 → 스크린샷/GIF

### 터치포인트 B: 블로그 기술 글 (/blog/[slug])
- **박준혁 경로**
- 문제 상황 → 고려한 대안 → 선택 이유 → 결과 + 회고

### 감정/생각
- 김서연: "의사결정 과정이 잘 정리되어 있네. 이 사람 인터뷰해보고 싶다"
- 박준혁: "점진적 마이그레이션을 선택한 이유가 합리적이네. 면접에서 이 부분을 더 파보자"

### 이탈 위험
- 케이스 스터디가 "했다" 나열이면 → 이력서와 똑같음 → 사이트를 본 보람 없음
- 블로그 글이 피상적이면 → "이 사람 깊이가 없구나" → 면접 안 부를 수 있음

### 핵심 목표
- "이 사람 면접에서 더 얘기해보고 싶다" → 면접 결정

---

## Stage 5: 판단/결정 (5분+)

### 터치포인트
- 사이트 떠난 후, 채용 시스템에서 판단 입력
- 또는 About 페이지에서 최종 프로필 확인

### 감정/생각
- 김서연: "사이트도 잘 만들었고, 프로젝트 성과도 구체적이야. 면접 잡자"
- 박준혁: "기술 글 수준이 괜찮고, 의사결정 과정이 논리적이야. 면접에서 아키텍처 질문 중심으로 가자"

### 이탈 위험
- 사이트를 닫은 후 기억에 남는 것이 없으면 → 다른 지원자에 묻힘
- **대응**: 각 페이지에 기억에 남는 수치/스토리 하나씩 배치

### 핵심 목표
- **면접 제안** (채용담당자)
- **합격 추천** (테크리드, 면접 후)

## 보너스 Journey: 동료 개발자 (이수진)

```
구글 검색 "vite 마이그레이션 실무"
  → /blog/webpack-to-vite-migration (SEO 유입)
    → 글 읽기 (3~7분)
      → "이 사람 다른 글도 볼까?" → 글 하단 관련 글
        → /blog (다른 글 2~3개 탐색)
          → / (랜딩) 또는 /about → "괜찮은 개발자네"
            → RSS 구독 또는 북마크
```

## 터치포인트별 KPI

| Stage | 시간 | KPI | 목표 |
|-------|------|-----|------|
| First Impression | 0~3초 | Bounce Rate | < 40% |
| Scanning | 3~30초 | Scroll Depth (랜딩) | > 60% |
| Scanning | 3~30초 | 프로젝트/블로그 클릭률 | > 30% |
| Deep Dive | 30초~5분 | 평균 체류시간 (케이스 스터디) | > 2분 |
| Deep Dive | 30초~5분 | 평균 체류시간 (블로그 글) | > 3분 |
| 판단 | 5분+ | 페이지 뷰 수 / 세션 | > 3 페이지 |

---

## 페이지별 역할 정리

| 페이지 | 주요 역할 | 주요 페르소나 | 핵심 전환 |
|--------|----------|-------------|----------|
| `/` (랜딩) | 3초 신뢰 + 탐색 유도 | 김서연 (채용담당자) | → /projects 또는 /blog 클릭 |
| `/projects` | 프로젝트 카드 스캔 + 수치 확인 | 김서연 | → /projects/[slug] 클릭 |
| `/projects/[slug]` | 깊이 확인 + 면접 결정 근거 | 김서연, 박준혁 | → 면접 결정 |
| `/blog` | 기술 글 목록 + 깊이 감지 | 박준혁 (테크리드) | → /blog/[slug] 클릭 |
| `/blog/[slug]` | 기술적 사고 확인 + 연쇄 탐색 | 박준혁, 이수진 | → 다른 글 클릭, 면접 질문 소재 |
| `/about` | 최종 프로필 확인 | 전체 | → 확신 강화 |

## 핵심 설계 원칙 (Journey에서 도출)

1. **3초 룰**: 랜딩은 Hero 한 줄 + 시각적 품질로 승부. 콘텐츠 과다 금지.
2. **수치 먼저**: 프로젝트 카드에 성과 수치를 제목/설명보다 먼저 보여주기
3. **깊이는 선택적**: 홈은 가볍게, 깊이는 서브페이지에서 (leeboa 패턴)
4. **경로 분리**: 채용담당자 → Projects 경로 / 테크리드 → Blog 경로 / 개발자 → 검색 유입
5. **이탈 방지 최우선**: LCP < 1.5초, 모바일 완벽 대응, 깨진 링크 제로
