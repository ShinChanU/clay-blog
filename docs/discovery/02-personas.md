# User Personas — claychanwoo.com 방문자

> ICP 분석을 기반으로 생성한 3개 페르소나
> 각 페르소나가 사이트에서 어떤 경험을 해야 하는지 구체적으로 정의

---

## Persona 1: "김서연" — 스타트업 채용담당자

### Demographics
- 31세, HR 매니저 / 피플팀
- 100명 규모 시리즈B SaaS 스타트업
- 기술 채용 경력 3년차
- 하루에 이력서 30~40개 검토, 한 이력서당 평균 45초 투자

### Primary Job-to-be-Done
> "이 지원자가 우리 팀에서 실제로 일할 수 있는 사람인지, 2분 안에 판단해서 면접 파이프라인에 올릴지 말지 결정하고 싶다"

- **빈도**: 매일, 서류 검토 시
- **맥락**: 원티드/사람인에서 지원이 들어오면 이력서 확인 → 링크 클릭 → 빠르게 판단

### Top 3 Pain Points

1. **"모르는 회사면 감이 안 온다"**
   - 다키클라우드, 에너지엑스 같은 이름만 보면 "어떤 수준의 일을 했는지" 상상이 안 됨
   - 영향: 불확실성 → 보수적 판단 → 서류 탈락 가능성↑

2. **"3~4년차가 가장 판단하기 어렵다"**
   - 주니어에게는 잠재력을, 시니어에게는 경험을 보면 되는데, 중간은 둘 다 애매
   - 이력서 내용이 과하면 "진짜 다 했나?" 의심, 적으면 "이것밖에 안 했나?" 실망

3. **"AI가 쓴 것 같은 이력서가 너무 많다"**
   - 기술 나열 + 완벽한 문장 → "이 사람이 직접 쓴 건가?" 신뢰 하락
   - 차별화 포인트가 없으면 다 비슷해 보임

### Top 3 Desired Gains

1. **즉각적 신뢰 신호**: 사이트를 열었을 때 "이 사람 괜찮다"는 직감을 주는 시각적 품질
2. **수치 기반 증거**: "MAU 156% 증가", "로딩 1/4 단축" 같은 정량 성과 — 이력서의 내용이 진짜인지 확인
3. **면접 질문 소재**: 케이스 스터디를 보면서 "이 부분 면접에서 더 물어봐야겠다"는 판단 근거

### Unexpected Insight
> 채용담당자는 사이트의 **"완성도" 자체**를 개발자 역량의 프록시로 사용한다.

잘 만든 개인 사이트 = "디테일을 챙기는 사람" = "일도 잘 하겠지"라는 무의식적 연결. 콘텐츠 깊이보다 **첫인상의 시각적 품질**이 서류 통과율에 더 큰 영향을 미침.

### Site Experience Design
- **랜딩 3초**: 깔끔한 Hero + 한줄 포지셔닝 → "프론트엔드 개발자구나" 즉시 인지
- **30초**: Featured Projects 카드에서 수치 성과 스캔 → "실적이 있는 사람이네"
- **2분**: 프로젝트 1개 상세 클릭 → 의사결정 과정 확인 → "면접에서 물어봐야지"
- **이탈 방지**: 로딩 느리면 바로 이탈. 모바일에서 깨지면 즉시 이탈.

## Persona 2: "박준혁" — 프론트엔드 테크리드

### Demographics
- 36세, 프론트엔드 챕터 리드
- 200명 규모 커머스 테크기업
- 개발 경력 9년, 프론트엔드 6년
- 주 2~3회 면접관으로 참여, 이력서 링크를 사전에 꼼꼼히 확인

### Primary Job-to-be-Done
> "이 지원자의 기술적 사고 수준과 성장 잠재력을 면접 전에 파악해서, 면접 시간을 효율적으로 쓰고 싶다"

- **빈도**: 주 2~3회, 면접 대상자 이력서 검토 시
- **맥락**: HR에서 서류 통과 리스트를 받으면 → 블로그/GitHub를 직접 확인 → 면접 질문 준비

### Top 3 Pain Points

1. **"기술 나열만 있으면 실력을 모르겠다"**
   - "React, TypeScript, Next.js, Zustand..." → 모든 3년차가 다 쓰는 목록
   - "왜 이 기술을 선택했고, 어떤 트레이드오프가 있었는지"가 없으면 판단 불가

2. **"튜토리얼 블로그는 실무 수준을 말해주지 않는다"**
   - "React useState 사용법" 같은 글 → 초보자 콘텐츠 → 시니어로 기대하기 어려움
   - 실무 문제 해결 과정이 담긴 글이어야 기술 수준 판단 가능

3. **"면접에서 뭘 물어봐야 할지 모르겠다"**
   - 이력서가 모호하면 면접 질문도 일반적이 되고, 좋은 답변을 끌어내기 어려움
   - 구체적인 프로젝트 스토리가 있어야 깊이 있는 질문 가능

### Top 3 Desired Gains

1. **의사결정 과정 확인**: "Webpack→Vite를 왜 점진적으로 했는지" → 이 사람의 사고 방식 이해
2. **아키텍처 감각 확인**: 시스템 설계 다이어그램, 컴포넌트 구조 결정 → 설계 능력 판단
3. **성장 방향 확인**: "사이드에서 실험 → 본업에 적용"하는 학습 루프 → 성장 잠재력 판단

### Unexpected Insight
> 테크리드는 **글의 "톤"**으로 협업 역량을 무의식적으로 판단한다.

"내가 했다, 내가 만들었다" vs "팀에서 논의했고, 내가 제안해서 검증했다" — 글 쓰는 톤에서 독선적인 사람인지 협업 가능한 사람인지를 읽어냄. 기술 블로그의 톤이 면접 전 인상을 결정함.

### Site Experience Design
- **블로그 진입**: 글 제목 리스트만 봐도 "이 사람이 어떤 수준의 문제를 다루는지" 파악
- **글 내용**: 문제 상황 → 고려한 대안들 → 선택한 이유 → 결과 + 회고 구조
- **프로젝트 상세**: 아키텍처 다이어그램 + 기술 선택 근거 + 수치 결과
- **핵심 경로**: /blog 글 제목 스캔 → 관심 글 1-2개 깊이 읽기 → /projects에서 실무 맥락 확인

## Persona 3: "이수진" — 동료 프론트엔드 개발자

### Demographics
- 28세, 프론트엔드 개발자 2년차
- 60명 규모 스타트업
- 기술 블로그 구독자, 개발 커뮤니티 활동 중
- 기술 문제 해결 시 구글 검색으로 블로그에 유입

### Primary Job-to-be-Done
> "내가 지금 겪고 있는 기술 문제의 실무적 해결책을 찾고, 더 나은 개발자가 되기 위한 인사이트를 얻고 싶다"

- **빈도**: 주 3~5회, 개발 중 막히는 부분 검색 시
- **맥락**: "Vite 마이그레이션 방법" 구글 검색 → 블로그 글에 직접 랜딩

### Top 3 Pain Points

1. **"공식 문서만으로는 실무 적용이 어렵다"**
   - 공식 문서는 이론/API 레퍼런스, 실무에서 "어떤 상황에서 어떻게 썼는지"가 없음

2. **"실무 경험이 담긴 한국어 기술 블로그가 부족하다"**
   - 영어 블로그는 많지만 한국 실무 환경(스타트업, B2B SaaS)에 맞는 콘텐츠가 적음

3. **"이 기술을 쓸지 말지 판단이 어렵다"**
   - FSD를 도입할까? Zustand vs Jotai? → 실제 도입 경험담이 있으면 판단에 큰 도움

### Top 3 Desired Gains

1. **실전 적용 사례**: 이론이 아닌 "실제로 해보니 이랬다"는 경험담
2. **의사결정 프레임워크**: "이런 상황이면 이걸 쓰고, 저런 상황이면 저걸 쓴다" 판단 기준
3. **커리어 롤모델**: "3년차에 이런 수준으로 성장할 수 있구나" 동기부여

### Unexpected Insight
> 동료 개발자는 블로그 글 하나에서 만족하면 **저자의 다른 글도 3~5개 연쇄 탐색**한다.

한 글이 좋으면 "이 사람 다른 글도 볼까?" → 프로필/소개 확인 → 다른 글 탐색 → 북마크/구독. 이 연쇄 탐색을 유도하는 UX(관련 글 추천, 시리즈 연결)가 체류시간과 재방문을 결정함.

### Site Experience Design
- **SEO 유입**: 검색 결과에서 제목+설명이 매력적 → 클릭
- **글 내부**: 실무 코드 예시 + 의사결정 맥락 → "이 사람 진짜 해봤네"
- **연쇄 탐색 유도**: 글 하단에 "관련 글" 또는 "같은 시리즈" 링크
- **프로필 확인**: /about 페이지에서 "나와 비슷한 연차인데 이만큼 했네" → 동기부여 + 팔로우

## 페르소나별 사이트 경로 요약

```
김서연 (채용담당자):
  이력서 링크 → / (랜딩) → Featured Projects 스캔 → /projects/[slug] 1개 확인 → 면접 결정

박준혁 (테크리드):
  이력서 링크 → /blog (글 목록) → 기술 글 2-3개 깊이 읽기 → /projects (실무 맥락) → 면접 질문 준비

이수진 (동료 개발자):
  구글 검색 → /blog/[slug] (개별 글) → 저자 프로필 확인 → /blog (다른 글 탐색) → 북마크/구독
```
